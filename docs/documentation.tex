\section{Introduction}
% =============================================================================

This document describes the design and implementation of a two-port
\textbf{software Ethernet switch} implemented in C\# as part of the PSIP
semester project.

The switch operates at \emph{OSI Layer~2} (Data Link layer).  It captures
raw Ethernet frames on two network interfaces using
\textbf{SharpPcap~6.3.1}~\cite{sharppcap} (backed by libpcap on
Linux/macOS or Npcap on Windows), learns source MAC addresses, forwards frames
to the correct output port, and tracks per-port traffic statistics.

\subsection{Scope}

The following functionality is covered in this document:

\begin{enumerate}[noitemsep]
  \item Frame reception, processing, and forwarding
        (Section~\ref{sec:forwarding})
  \item MAC address table management (Section~\ref{sec:mac})
  \item Traffic filtering design (Section~\ref{sec:filtering})
  \item Traffic statistics (Section~\ref{sec:stats})
  \item Cable disconnection and replacement handling
        (Section~\ref{sec:cable})
  \item Optional protocol extensions: Syslog, CDP, RESTCONF
        (Section~\ref{sec:extensions})
\end{enumerate}

\subsection{Thread and task model}

\begin{description}[noitemsep]
  \item[Capture callbacks] SharpPcap dispatches \texttt{OnPacketArrival}
    events from its internal per-device background threads (one per port).
    Both callbacks share a single \texttt{lock} object before touching the
    MAC table or statistics.
  \item[MAC expiry task] A background \texttt{async Task} wakes every
    5~seconds and removes entries whose age exceeds the configured TTL.
  \item[Web server thread] The \texttt{HttpListener} thread handles HTTP
    requests from the administrator's browser.
\end{description}

\subsubsection{Key design decisions}

\begin{enumerate}[noitemsep]
  \item \textbf{Unknown destination} -- If the destination MAC is absent from
    the table the frame is flooded to the \emph{opposite} port (standard
    unicast-unknown behaviour).
  \item \textbf{Same-port guard} -- If the MAC table would route the frame
    back to its arriving port (e.g.\ after a topology change), the switch
    overrides to the opposite port to prevent the frame being dropped.
  \item \textbf{Promiscuous mode} -- Both pcap devices are opened with
    \texttt{DeviceModes.Promiscuous} so all Ethernet frames on the medium
    are captured, regardless of destination MAC.
  \item \textbf{NoCaptureLocal} -- The \texttt{NoCaptureLocal} flag prevents
    the capture device from seeing frames the bridge itself injected,
    eliminating forwarding loops without any extra outgoing-frame check.
\end{enumerate}

\subsection{Protocol detection}
\label{sec:detection}

After extracting the MACs, the switch inspects EtherType and higher-layer
headers to classify the frame.  Table~\ref{tab:protos} lists all recognised
protocols.

\begin{table}[H]
\centering
\caption{Protocol detection logic.}
\label{tab:protos}
\begin{tabularx}{\linewidth}{llX}
\toprule
\textbf{Protocol} & \textbf{Detection condition} & \textbf{Notes} \\
\midrule
Ethernet~II & always & Base layer; every frame is counted. \\
ARP         & EtherType = 0x0806 & Address Resolution Protocol. \\
IP          & EtherType = 0x0800, frame $\geq$ 34\,B & IPv4 only. \\
ICMP        & IP proto = 1 & Echo / ping traffic. \\
TCP         & IP proto = 6, frame $\geq$ 14+IHL+20\,B & \\
HTTP        & TCP port\,80 + payload starts with
              \texttt{GET}/\texttt{POST}/\texttt{PUT}/\texttt{DELETE}/
              \texttt{HEAD} or \texttt{HTTP/} & Plaintext HTTP only. \\
UDP         & IP proto = 17 & \\
\bottomrule
\end{tabularx}
\end{table}

Detection is hierarchical: an HTTP frame also increments TCP, IP, and
Ethernet~II counters.

\subsubsection{C\# implementation excerpt}

\begin{lstlisting}[language=CSharp, caption={Protocol detection (Switch.cs).}]
ushort ethertype = (ushort)((frame[12] << 8) | frame[13]);
if (ethertype == 0x0806) { found.Add("arp"); return found; }
if (ethertype != 0x0800 || frame.Length < 34) return found;
found.Add("ip");
byte proto = frame[23];
if (proto == 1) { found.Add("icmp"); return found; }
if (proto == 6 && frame.Length >= l4Start + 20) {
    found.Add("tcp");
    // ... HTTP payload check on port 80 ...
    return found;
}
if (proto == 17) found.Add("udp");
\end{lstlisting}

\section{MAC Address Table Management}
\label{sec:mac}
% =============================================================================

\subsection{Data structure}

The MAC table is an in-memory \texttt{Dictionary<string, MacEntry>} keyed on
the colon-separated 17-character MAC string
(e.g.\ \texttt{aa:bb:cc:dd:ee:ff}).  Each entry stores:

\begin{center}
\begin{tabular}{lll}
\toprule
\textbf{Field}    & \textbf{Type}     & \textbf{Meaning} \\
\midrule
\texttt{Port}     & \texttt{int}      & Port (1 or 2) where the host was last seen. \\
\texttt{LastSeen} & \texttt{DateTime} & UTC timestamp of the last frame from this MAC. \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Adding an entry (MAC learning)}

Every processed frame causes the \emph{source} MAC to be examined:

\begin{itemize}[noitemsep]
  \item If the MAC is \textbf{not} in the table a new \texttt{MacEntry} is
    created with the current port and timestamp.
  \item If the MAC \textbf{is} already present the existing entry's port and
    timestamp are updated in place (handles port migration).
\end{itemize}

MAC learning is a side-effect of forwarding; there is no separate ``add''
API.

\subsection{Removing an entry (TTL expiry)}

Each entry has a configurable time-to-live (default: \textbf{300\,s}).
A background \texttt{async Task} checks all entries every 5\,seconds and
removes those whose age exceeds the TTL:

\begin{lstlisting}[language=CSharp, caption={MAC expiry background task (Switch.cs).}]
private async Task ExpiryLoopAsync()
{
    while (!_cts.Token.IsCancellationRequested)
    {
        await Task.Delay(ExpiryCheckIntervalMs, _cts.Token);
        PurgeExpiredEntries();
    }
}

public void PurgeExpiredEntries()
{
    lock (_lock)
    {
        var expired = _macTable
            .Where(kv => kv.Value.IsExpired(MacTtlSeconds))
            .Select(kv => kv.Key).ToList();
        foreach (var mac in expired)
            _macTable.Remove(mac);
    }
}
\end{lstlisting}

Entries can also be cleared immediately via \texttt{ClearMacTable()} (the
``Clear MAC table'' button in the web GUI).

\subsection{Updating an entry (cable replacement / port migration)}
\label{sec:mac_update}

When a host moves to a different port, the next frame it sends arrives on the
new port with the same source MAC.  The switch updates the existing entry
atomically inside the same \texttt{lock}:

\begin{lstlisting}[language=CSharp, caption={In-place MAC entry update (Switch.cs).}]
if (_macTable.TryGetValue(srcMac, out var existing))
{
    existing.Port = inPort;            // redirect to new port
    existing.LastSeen = DateTime.UtcNow; // reset TTL
}
else
{
    _macTable[srcMac] = new MacEntry(inPort);
}
\end{lstlisting}

The switch converges within a single frame -- no flooding is required after
the update.

\subsection{Manual TTL configuration}

The TTL can be changed at runtime via \texttt{SetMacTtl()} or the web GUI
without restarting the switch.

\subsection{Current state}

The current prototype does not implement a configurable filter.  All frames
passing the 14-byte minimum length check are forwarded.

\subsection{Proposed filter architecture}

A filter subsystem can be inserted as a pipeline stage between reception and
the forwarding decision (Figure~\ref{fig:filter}).

\subsubsection{Filter rule model}

Each rule is evaluated in priority order and specifies:

\begin{center}
\begin{tabular}{lll}
\toprule
\textbf{Field}   & \textbf{Example}                   & \textbf{Notes} \\
\midrule
Priority         & 10, 20, \ldots                     & Lower = higher priority. \\
Match: src MAC   & \texttt{aa:bb:cc:dd:ee:ff} or any  & \\
Match: dst MAC   & broadcast, multicast group          & \\
Match: EtherType & 0x0800 (IP), 0x0806 (ARP)           & \\
Match: in-port   & 1, 2, any                           & \\
Action           & permit / deny                       & Default: permit. \\
\bottomrule
\end{tabular}
\end{center}

\subsubsection{C\# interface sketch}

\begin{lstlisting}[language=CSharp, caption={Proposed filter interface.}]
public interface IFrameFilter
{
    /// <summary>Returns true if the frame should be forwarded.</summary>
    bool ShouldForward(int inPort, byte[] frame);
}

public sealed class AclFilter : IFrameFilter
{
    private readonly IReadOnlyList<AclRule> _rules;
    public bool ShouldForward(int inPort, byte[] frame)
    {
        foreach (var rule in _rules)
            if (rule.Matches(inPort, frame))
                return rule.Action == AclAction.Permit;
        return true; // default permit
    }
}
\end{lstlisting}

\texttt{Switch.ProcessFrame} would call \texttt{filter.ShouldForward()}
before the forwarding decision and silently discard the frame (without
updating statistics) when it returns \texttt{false}.

\subsubsection{Layer-3 / Layer-4 filtering}

Because the switch already classifies frames into Ethernet~II, ARP, IP, TCP,
UDP, ICMP, and HTTP (Section~\ref{sec:detection}), extended rules could also
inspect:

\begin{itemize}[noitemsep]
  \item IPv4 source / destination address (bytes 26--29 / 30--33)
  \item TCP/UDP port numbers (bytes 34--35 / 36--37)
  \item ICMP type (byte 34)
\end{itemize}

This is equivalent to a Cisco IOS \emph{extended ACL}.

\section{Traffic Statistics}
\label{sec:stats}
% =============================================================================

\subsection{Counters maintained}

Per-port RX and TX counters are updated for every frame:

\begin{center}
\begin{tabular}{lll}
\toprule
\textbf{Counter}   & \textbf{Dir.} & \textbf{Description} \\
\midrule
\texttt{RxFrames}  & RX & Total Ethernet frames received on this port. \\
\texttt{TxFrames}  & TX & Total Ethernet frames transmitted on this port. \\
\texttt{RxPdus[p]} & RX & PDU count for protocol $p$ on receive. \\
\texttt{TxPdus[p]} & TX & PDU count for protocol $p$ on transmit. \\
\bottomrule
\end{tabular}
\end{center}

where $p \in \{\texttt{ethernet\_ii},\ \texttt{arp},\ \texttt{ip},\
\texttt{tcp},\ \texttt{udp},\ \texttt{icmp},\ \texttt{http}\}$.

\subsection{Update logic}

Statistics are updated \emph{atomically} with the forwarding decision inside
the same \texttt{lock} block:

\begin{lstlisting}[language=CSharp, caption={Statistics update (Switch.cs).}]
var protocols = DetectProtocols(frame);
_stats[inPort].IncrementRx(protocols);   // RX on arriving port
_stats[outPort].IncrementTx(protocols);  // TX on forwarding port
\end{lstlisting}

A frame arriving on port~1 and forwarded to port~2 increments port~1 RX and
port~2 TX.  Protocol counting is hierarchical: an HTTP frame increments
\texttt{http}, \texttt{tcp}, \texttt{ip}, and \texttt{ethernet\_ii}
simultaneously.

Figure~\ref{fig:stats_flow} shows the exact points in the pipeline where
counters are updated.

\section{Cable Disconnection and Replacement}
\label{sec:cable}
% =============================================================================

\subsection{Cable removal -- detection and handling}

The software switch is a \emph{passive} bridge: it only processes frames it
receives and issues no probes.  It therefore cannot detect a physical
link-down event on its own.

\subsubsection{Scenario: cable pulled on port $N$}

\begin{enumerate}[noitemsep]
  \item No more frames arrive on port~$N$.
  \item All MAC table entries last seen on port~$N$ survive until their TTL
    expires (default 300\,s).
  \item During the TTL window the switch may still forward frames
    \emph{toward} port~$N$; the OS silently drops them because the
    interface is link-down.
  \item After the TTL the entries are purged; subsequent frames to those
    hosts are flooded to the opposite port.
\end{enumerate}

\subsubsection{Improvement: link-state monitoring}

A production-grade extension would subscribe to
\texttt{NetworkChange.NetworkAddressChanged} (.NET) and immediately flush MAC
entries for the affected port on link-down:

\begin{lstlisting}[language=CSharp, caption={Proposed link-down handler.}]
NetworkChange.NetworkAddressChanged += (_, _) =>
{
    foreach (var iface in NetworkInterface.GetAllNetworkInterfaces())
    {
        if (iface.OperationalStatus == OperationalStatus.Down)
        {
            int port = GetPortForInterface(iface.Name);
            if (port != 0)
                _switch.FlushEntriesForPort(port); // new API
        }
    }
};
\end{lstlisting}

\subsection{Cable replacement (port migration)}

When a host moves to a different port the first frame it transmits triggers
an in-place update of its MAC table entry
(Section~\ref{sec:mac_update}).
Figure~\ref{fig:cable} illustrates the convergence timeline.

\section{Optional Extensions: Syslog, CDP, RESTCONF}
\label{sec:extensions}
% =============================================================================

The following protocols are not implemented in the current prototype but are
analysed below as potential extensions.

\subsection{Syslog (RFC~5424)}

Syslog~\cite{rfc5424} provides a standard mechanism for logging system
messages to a central collector over UDP port~514 (or TCP port~6514).
The switch could emit Syslog messages for:

\begin{itemize}[noitemsep]
  \item New MAC learned (Severity: Informational)
  \item MAC migrated to new port (Severity: Notice)
  \item MAC entry expired (Severity: Debug)
  \item Frame discarded by ACL filter (Severity: Warning)
  \item Link-down detected (Severity: Critical)
\end{itemize}

A Syslog message has the format:\\
\texttt{<PRI>VERSION SP TIMESTAMP SP HOSTNAME SP APP-NAME SP PROCID SP MSGID
SP STRUCTURED-DATA SP MSG}

where \texttt{PRI} = (Facility $\times$ 8 + Severity).

Figure~\ref{fig:syslog} shows a Syslog message flow between switch and
collector.

\centering

  \node[graybox,  right=of sw] (col)  {Syslog Collector\\(UDP 514)};

  \draw[arr] (sw.east) -- node[above, font=\small,
    align=center]
    {\texttt{<134>1 2025-03-10T12:00:00Z switch - - MAC learned}}
    (col.west);

\end{tikzpicture}
\caption{Syslog message flow from the switch to a collector.}
\label{fig:syslog}
\end{figure}

\subsection{CDP -- Cisco Discovery Protocol}

CDP~\cite{cdp} is a proprietary Cisco Layer-2 protocol.  Cisco devices
periodically transmit CDP advertisements as multicast frames to
\texttt{01:00:0c:cc:cc:cc} using SNAP encapsulation (LLC DSAP/SSAP = 0xAA,
OUI = 0x00000C, PID = 0x2000).

Each frame carries TLV (Type-Length-Value) records:

\begin{center}
\begin{tabular}{lll}
\toprule
\textbf{Type} & \textbf{Name}      & \textbf{Content} \\
\midrule
0x0001 & Device ID     & Hostname of the sending device. \\
0x0002 & Addresses     & Layer-3 address(es) of the device. \\
0x0003 & Port ID       & Interface name of the sending port. \\
0x0004 & Capabilities  & Bitmask (Router, Switch, Host, \ldots). \\
0x0005 & Software Ver. & IOS version string. \\
0x0006 & Platform      & Hardware platform string. \\
\bottomrule
\end{tabular}
\end{center}

The switch could passively decode CDP frames and log the advertised device
information without joining the Cisco CDP domain.

Figure~\ref{fig:cdp} illustrates the CDP frame structure.

\subsection{RESTCONF (RFC~8040)}

RESTCONF~\cite{rfc8040} is a REST-based protocol for managing YANG-modelled
network devices over HTTPS.  A RESTCONF interface would expose the switch's
MAC table and statistics as REST resources:

\begin{center}
\begin{tabular}{lll}
\toprule
\textbf{Method} & \textbf{URL}                             & \textbf{Action} \\
\midrule
GET    & \texttt{/restconf/data/switch:mac-table}     & Retrieve all MAC entries. \\
DELETE & \texttt{/restconf/data/switch:mac-table}     & Clear the table. \\
PATCH  & \texttt{/restconf/data/switch:mac-table/ttl} & Set TTL. \\
GET    & \texttt{/restconf/data/switch:statistics}    & Retrieve port counters. \\
DELETE & \texttt{/restconf/data/switch:statistics}    & Reset counters. \\
\bottomrule
\end{tabular}
\end{center}

The current \texttt{WebGui} HTTP API is a lightweight analogue of RESTCONF;
a formal implementation would require a YANG data model and a RESTCONF
framework library.

\section*{Conclusion}
\addcontentsline{toc}{section}{Conclusion}
% =============================================================================

The prototype demonstrates the essential functionality of an Ethernet
Layer-2 switch: MAC learning, unicast forwarding, unknown-destination
flooding, hierarchical protocol-aware statistics, time-based MAC table expiry,
and a web-based management interface.

Open items for a production-grade implementation include:

\begin{itemize}[noitemsep]
  \item Link-state monitoring with immediate port flush on link-down.
  \item Configurable ACL-based traffic filtering.
  \item Syslog integration for structured event logging.
  \item RESTCONF / YANG API for programmatic management.
\end{itemize}
